# YESPC OS 現代化 UI 改版藍圖 v1

本文件以 YESPC OS v5.9.9 的實際代碼結構為依據，規劃每個模組從「現況」到「新版」的 UI 改版方向。  
目標是：**功能不減、操作不變、畫面變得更專業、結構變得更容易被 AI 工具維護。**

---

## 一、設計原則

### 1.1 維持手機第一

YESPC OS 目前就是以手機為主要操作介面，未來新版 UI 同樣維持手機優先設計。  
所有模組的主要操作流程，包含新增交易、建立維修單、快速盤點，都必須能在手機上一手操作完成，不能因為「UI 變漂亮」而增加操作步數。

### 1.2 資訊層級明確

目前許多模組把 KPI 摘要、表單、清單、歷史紀錄全部堆在同一個垂直捲動頁面，視覺上不容易判斷「現在在哪個區塊」。  
新版 UI 必須把每個模組的資訊切成明確層級：

- **第一層**：KPI 摘要 / 今日狀態（最重要、最顯眼）
- **第二層**：快速操作區 / 主表單（最常用的動作）
- **第三層**：清單 / 歷史（次要、可縮合或分頁）
- **第四層**：詳情抽屜 / 展開面板（點開才看）

### 1.3 顏色使用更克制

目前系統每個模組都有自己的主色（SalesModule 藍、RepairModule 綠、CrmModule 玫瑰、StockModule 琥珀），整體偏活潑。  
新版建議改成：**以白色 / 淺灰為主要底色，各模組用一個主色作為「強調點」而非全面著色**，這樣會更有商業感。

### 1.4 元件標準化

目前 KPI 卡、表單輸入框、清單卡片、按鈕、標籤在每個模組裡都是「各自為政」的實作方式。  
新版應定義：KPI 卡長什麼樣、清單卡長什麼樣、抽屜詳情長什麼樣、主要按鈕長什麼樣，做成可重複使用的元件邏輯。

### 1.5 不要破壞現有邏輯

UI 改版不等於邏輯改版。  
所有資料讀寫邏輯（Firebase 連線、safeDelete、CRM 自動帶入、日結流程）在本版改版中**完全不動**，只改視覺與資訊排版方式。

## 二、導覽系統改版

### 2.1 現況

目前系統有兩層導覽：

- **頂部 Header**：顯示系統名稱、版本號、可切換 Owner 權限
- **底部 Tab Bar**：7 個功能按鈕（Portal、Board、Sales、Repair、CRM、Stock、Admin），加上 PortalModule 的 2x3 宮格選單

問題點：底部 Tab Bar 有 7 個項目，加上 Portal 宮格，使用者進入系統後要先點一次 Portal、再點一次模組，多了一個步驟。

### 2.2 新版建議

**方案 A（保守）：維持底部 Tab + 改善視覺**

```
底部 Tab Bar（維持 7 項，改善圖示與文字大小）：
[ ⊞ 總覽 ]  [ 📋 待辦 ]  [ 💰 帳務 ]  [ 🔧 維修 ]  [ 👤 客戶 ]  [ 📦 庫存 ]  [ ⚙️ 設定 ]
```

- 把「Portal 宮格」改成「總覽 Dashboard」，第一眼看到的是今日摘要，不再是功能入口
- Tab 圖示統一換成 Lucide Icon 系列，取代目前的 Emoji
- Tab 文字統一 10px，只有 active 狀態才有底部指示線
- Header 高度從目前的 `py-4` 縮減為 `py-2`，節省空間

**方案 B（積極）：側邊導覽（桌機版）+ 底部 Tab（手機版）**

```
手機版：底部 Tab Bar，同方案 A
桌機版：
┌────────────────────────────────────────────────┐
│  [YESPC OS]  ▶  v5.9.9      [今日 2026/04/16]  │ ← 頂部 Header
├──────────┬─────────────────────────────────────┤
│ ⊞ 總覽  │                                      │
│ 📋 待辦  │    （模組主內容區）                  │
│ 💰 帳務  │                                      │
│ 🔧 維修  │                                      │
│ 👤 客戶  │                                      │
│ 📦 庫存  │                                      │
│ ⚙️ 設定  │                                      │
└──────────┴─────────────────────────────────────┘
```

**建議採用方案 A**，因為：
- 不需要加 CSS breakpoint 邏輯
- 現有架構不用改導覽 state 系統
- 你的主要使用情境還是手機

## 三、各模組改版細節

### 3.1 PortalModule → 改為「今日總覽 Dashboard」

**現況問題**：目前 Portal 是 2x3 功能宮格，像個 App 主畫面，對於已經熟悉系統的使用者來說每次都要多點一下。

**新版方向**：把 Portal 改成「今日狀態面板」，用戶進入系統第一眼就看到今天的關鍵數字。

```
┌─────────────────────────────────────────┐
│  今日   2026/04/16 (週四)               │
├────────────┬────────────┬───────────────┤
│  今日收入   │  今日毛利  │  現金庫存     │
│  $12,500   │  $3,200    │  $8,200       │
├────────────┴────────────┴───────────────┤
│  📋 待辦未完成   3 項    ──────────►    │
│  🔧 維修進行中   2 件    ──────────►    │
│  📦 庫存低警戒   1 項    ──────────►    │
└─────────────────────────────────────────┘
```

**改版要點**：
- 今日 KPI 三格：今日收入、今日毛利、現金庫存（維持現有計算邏輯）
- 捷徑列：顯示「未完成待辦數」、「維修進行中數」、「庫存低警戒數」，點擊直接跳對應模組
- 底部快捷入口：「+ 新增交易」、「+ 新增維修」兩個常用快捷按鈕

**實作難度**：低，只需重新排版，邏輯沿用。

### 3.5 StockModule 庫存模組

**現況問題**：目前是卡片清單，每張卡片有品名、分類、成本、利潤、數量增減按鈕。

- 無法快速搜尋特定品項

**新版資訊架構**：

```
StockModule 新版 ─ 搜尋列 + 分類篩選 + 清單
```

**展開後出現的編輯列**：

```
  [品名輸入] [分類] [供應商]
```

**實作難度**：低

## 四、共用 UI 元件規格

### 4.1 Bottom Sheet（底部滑出面板）

所有模組的「新增」、「編輯」動作統一用 Bottom Sheet 呈現，不再是嵌入在頁面中的表單。

```css
.bottom-sheet {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  background: white;
  border-radius: 1.5rem 1.5rem 0 0;
  padding: 1.5rem;
  transform: translateY(100%);
  transition: transform 0.3s cubic-bezier(0.16, 1, 0.3, 1);
  z-index: 100;
  max-height: 90vh;
  overflow-y: auto;
}

.bottom-sheet.active {
  transform: translateY(0);
}

.bottom-sheet-overlay {
  position: fixed;
  inset: 0;
  background: rgba(0,0,0,0.4);
  z-index: 99;
  opacity: 0;
  pointer-events: none;
  transition: opacity 0.3s ease;
}

.bottom-sheet-overlay.active {
  opacity: 1;
  pointer-events: all;
}
```

**觸發方式**：用 `sheetOpen` state 控制 `active` class 的切換。  
**關閉方式**：點擊 overlay、點擊右上角 ✕、或輕掃向下。

### 4.2 KPI 卡

```
┌──────────────────┐
│  標籤文字        │  ← 10px 灰色
│  $28,500         │  ← 24px 深色粗體
│  ↑ 較昨日 +12%   │  ← 12px 輔助說明（選填）
└──────────────────┘
```

KPI 卡固定 `border-radius: 1rem`，底色統一為白色，陰影 `shadow-sm`，不要用彩色底色。

```
│  副標題文字（12px 灰）                       │
│                              金額（右對齊）  │
│                              狀態標籤       │
```

## 八、與重構路線圖的關係

本版 UI 改版（v1）的所有改動，都在現有的 CDN + createElement + 單檔架構下進行。  
這樣做的好處是：你可以馬上看到改版效果、馬上使用、不需要任何環境設定。

但這些改版也是在為未來的架構遷移做準備：

- **Bottom Sheet 元件**：未來遷移到 React + Vite 後，就是第一個 `BottomSheet.jsx`
- **模組內 subTab**：未來就是每個模組的 Tab 元件
- **KPI 卡規格**：未來就是 `KpiCard.jsx`
- **清單卡片規格**：未來就是 `ListCard.jsx`

也就是說，你在這個階段定義好元件的視覺與行為標準，等到真正遷移架構時，AI 工具就能直接依照這份藍圖把每個元件寫成正式的 JSX 元件，不需要再猜你要什麼樣子。
